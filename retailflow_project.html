Project Overview

RetailFlow Group is a fictional retail holding company that operates three independent store brands, each with its own OLTP systems, event formats, and data quality characteristics. The project simulates the real-world challenge of consolidating heterogeneous sources into a single medallion-architecture data lakehouse (MinIO + Delta Lake), applying entity resolution to produce global user and product identities, and ultimately training a purchase-propensity model.

The three ingestion patterns — Kafka streaming, Spark batch DB extract, and file drops — are not distributed artificially. Each store uses the pattern that fits its maturity level, which is exactly how this looks in production at a real company post-acquisition.

Core challenge

Entity Resolution

The same customer shops all three stores. Each store assigns its own user ID. Silver layer must stitch identities via email hash, phone hash, and probabilistic matching.

Core challenge

Schema Heterogeneity

Same concepts, different shapes: camelCase vs snake_case, UUID vs integer IDs, flat vs nested price structures, boolean vs 0/1 flags — all across three stores.

Core challenge

Order ID Collisions

All three stores use sequential integer order IDs. order_id=10042 exists in all three. Bronze must namespace with store_id before deduplication.

ML goal

Purchase Propensity

Gold layer features feed a cross-store model predicting purchase probability per user per session. Served via FastAPI with real-time feature lookup from the feature store.

Bronze → Silver Normalisation Checklist

  • DEDUPDeduplicate Kafka events on event_id using watermark — client retries produce duplicates in both alpha and beta.
  • NULLuser_id is null for ~42% of clickstream (anonymous) and ~12–18% of orders (guest checkout). Never drop — use anonymous_id for stitching.
  • FORMATPrefix all order_ids with store_id ("alpha_10042", "beta_10042", "gamma_10042") — sequential integers collide across all three stores.
  • SCHEMAReconcile Kafka alpha v2.1 vs beta v1.3: "items" → "lineItems", "total_amount" → "totalPrice", missing tax_amount, discount_pct fraction vs percent.
  • PIIHash guest_email before Silver. Mask ip_address when gdpr_consent=false. Exclude order notes (free text) entirely.
  • FORMATNormalise all timestamps to UTC. gamma file drops use Europe/Warsaw or Europe/Berlin local time. Some beta Mongo docs use date-only strings.
  • TYPECoerce legacy boolean representations: active=0/1 (gamma CSV, beta legacy Mongo), gdprConsent=null → false, status "zrealizowane"→"completed" etc.
  • SCHEMAFlatten MongoDB prices array to per-region rows. Legacy docs have flat "price" float — synthesise single-region prices array before joining.
  • NULLrefund_amount key entirely absent (not null) in ~30% of gamma returns JSON. Always use .get("refund_amount", 0.0) — KeyError will crash the job.
  • TYPECoerce gamma returns items to always be an array — WMS sends a single object for single-item returns instead of a one-element array.
  • FORMATDetect and convert gamma CSV encoding: try UTF-8, fall back to Windows-1250. Strip BOM. Skip empty rows and summary rows at end of file.
  • FORMATNormalise gamma promo discount_value: detect fraction (≤1.0) vs percent (>1.0) and convert to consistent 0–100 scale.
  • SCHEMADeduplicate and lowercase MongoDB tags array: ["hoodie","Hoodie","HOODIE"] → ["hoodie"].
  • FORMATValidate and filter broken image URLs in MongoDB products before Silver. Log broken URLs as data quality metric.
  • SCHEMAMap warehouse-specific reason_code enums to canonical return reasons across WH-PL-01 and WH-DE-01.
  • NULLTreat MongoDB gdprConsent=null as false (pre-GDPR legacy profiles). Never infer consent from absence.
  • DEDUPEntity resolution (Silver → Gold): stitch global_user_id via email_hash, then phone_hash, with confidence_score. Stitch global_product_id via EAN barcode, then name similarity.
  • FORMATNormalise all currency amounts to PLN equivalent using daily exchange rate snapshot stored in MinIO. alpha/beta orders may be in EUR or GBP.

Build Plan — 8 Weeks

Week 1–2
Infrastructure + Data Generator
Docker Compose with all services. Python generator producing realistic events and DB records for all three stores. Kafka topics + Schema Registry + Avro schemas.
Week 2–3
Bronze Ingestion — All Three Patterns
PySpark Structured Streaming (Kafka → Bronze). Spark JDBC incremental extract (Postgres → Bronze). Spark Mongo connector (MongoDB → Bronze). File watcher (gamma drops → Bronze).
Week 3–4
Bronze → Silver + Great Expectations
All normalisation from the checklist. Schema reconciliation, type coercion, PII hashing, timestamp UTC conversion, ID namespacing. GX quality gates on every Silver table.
Week 4–5
Entity Resolution
Global user identity stitching via email/phone hash with confidence scoring. Global product deduplication via EAN barcode and name similarity (TF-IDF or Levenshtein).
Week 5–6
Gold Layer + dbt
unified_orders, global_users, global_products, user_features (cross-store LTV, category affinities, cart abandonment rate). dbt models with tests and lineage docs.
Week 6–7
ML Pipeline — Feast + MLflow
Feature store setup with Feast. Training dataset construction. Purchase propensity model (LightGBM via PySpark pandas UDF). MLflow experiment tracking and model registry.
Week 7–8
Serving + Observability + Docs
FastAPI /predict endpoint pulling from Feast + MLflow. Trino setup for ad-hoc SQL. Grafana dashboards. README, architecture diagram, and a recorded demo walkthrough.
